# プロンプト更新提案テンプレート

このテンプレートは、learnings.mdに蓄積された学び（weight 3以上）をプロンプトに反映する際に使用します。

---

## 📝 基本テンプレート

```markdown
# プロンプト更新提案

**提案日**: YYYY-MM-DD
**対象エージェント**: [エージェント名]
**対象プロンプト**: [プロンプトファイル名.md]
**提案理由**: [learning_id]（[学びのタイトル]）が weight [N] に到達

---

## 📊 該当する学び

### [learning_id]: [学びのタイトル]
- **weight**: [N]
- **カテゴリ**: [カテゴリ名]
- **フィードバック回数**: [N]回

**学んだこと**:
- [具体的な学びの内容1]
- [具体的な学びの内容2]
- [具体的な学びの内容3]

---

## 🔧 提案する変更

### Before (現在のプロンプト)

```markdown
[現在のプロンプトの該当セクション]
```

### After (提案)

```markdown
[更新後のプロンプトの該当セクション]
```

**変更のポイント**:
- [何をどう変更したか]
- [なぜこの変更が必要か]
- [どのような効果が期待できるか]

---

## ✅ 期待される効果

- [効果1: エージェントが自然と〜を意識]
- [効果2: learnings.mdを毎回読まなくても〜]
- [効果3: 同じミスを繰り返さない]

---

## 🎯 承認後のアクション

承認された場合、以下を実行します:

1. プロンプト（[ファイル名]）を更新
2. learnings.mdの該当エントリの status を `resolved` に変更
3. 変更履歴をプロンプトに記録
4. resolved_at と reflected_in フィールドを更新
```

---

## 🎯 使い方

### 1. トリガー条件

プロンプト更新提案を作成すべきタイミング:

| 条件 | アクション |
|------|-----------|
| **weight 5以上** | **即座に更新提案作成** |
| weight 3-4 | 月次レビューで検討 |
| weight 1-2 | 様子見 |

### 2. 更新すべき学び vs すべきでない学び

#### ✅ 更新すべき学び

- **普遍的な学び**
  - 例: 「文字数は135〜145字に厳守する」
  - どのタスクでも適用できる

- **基礎能力に関する学び**
  - 例: 「ターゲット設定を具体的に」
  - エージェントの核となるスキル

- **繰り返し指摘される学び**
  - weight 3以上になった学び
  - エージェントが定着させるべき

#### ❌ 更新すべきでない学び

- **プロジェクト固有の学び**
  - 例: 「今回のキャンペーンでは〜」
  - 特定のプロジェクトにのみ関連

- **一時的な学び**
  - 例: 「今週のトレンドは〜」
  - すぐに古くなる情報

- **ユーザー固有の学び**
  - 例: 「[ACCOUNT_OWNER]さんは〜が好き」
  - `shared/knowledge/user-profile.md` に記録すべき

### 3. 提案の書き方ポイント

#### Before/After を明確に

- 現在のプロンプトの該当箇所を **そのまま引用**
- 変更後のプロンプトを **具体的に記述**
- 変更部分を **太字** で強調

#### 期待される効果を具体的に

- 「エージェントが〜を意識するようになる」だけでなく
- 「これにより〜というミスがなくなる」まで書く

#### 変更のポイントを説明

- なぜこの場所に追加するのか
- なぜこの表現にしたのか
- 他の選択肢と比較してどうか

## 📋 エージェント別の例

### ECHO (X投稿スペシャリスト)

```markdown
# プロンプト更新提案

**提案日**: 2026-01-10
**対象エージェント**: ECHO
**対象プロンプト**: post-creator.md
**提案理由**: learning_echo_002（文字数厳守）が weight 5 に到達

---

## 📊 該当する学び

### learning_echo_002: 文字数厳守（140字前後）
- **weight**: 5
- **カテゴリ**: 品質管理
- **フィードバック回数**: 5回

**学んだこと**:
- 140字前後（135〜145字）を厳守する
- 短すぎると情報不足、長すぎると読まれない
- 100字以下は密度が薄くなりがち

---

## 🔧 提案する変更

### Before (現在のプロンプト)

```markdown
## 制約
- X投稿の基本ルールに従う
- 簡潔で読みやすい文章
```

### After (提案)

```markdown
## 制約
- X投稿の基本ルールに従う
- **文字数は135〜145字の範囲に厳守する**
  - 140字を目安に、前後5字以内に収める
  - 100字以下は避ける（密度が薄くなる）
  - 150字以上は避ける（読まれにくい）
- 簡潔で読みやすい文章
```

**変更のポイント**:
- 「制約」セクションに文字数の具体的な範囲を明記
- 上限・下限の理由も併記することで、意図が明確に
- 目安（140字）と許容範囲（±5字）を分けて記述

---

## ✅ 期待される効果

- エージェントが自然と文字数を意識するようになる
- learnings.mdを毎回読まなくても基礎能力として定着
- 文字数超過/不足のミスが発生しなくなる

---

## 🎯 承認後のアクション

承認された場合、以下を実行します:

1. プロンプト（post-creator.md）を更新
2. learnings.mdの learning_echo_002 の status を `resolved` に変更
3. 変更履歴をプロンプトに記録
4. resolved_at: 2026-01-10, reflected_in: prompts/post-creator.md を追加
```

### SAGE (note記事スペシャリスト)

```markdown
# プロンプト更新提案

**提案日**: 2026-01-15
**対象エージェント**: SAGE
**対象プロンプト**: article-creator.md
**提案理由**: learning_sage_001（導入部は3段落以内）が weight 4 に到達

---

## 📊 該当する学び

### learning_sage_001: 導入部は3段落以内に収める
- **weight**: 4
- **カテゴリ**: コンテンツ作成
- **フィードバック回数**: 4回

**学んだこと**:
- 導入部が長いと離脱される
- 3段落（300字程度）で問題提起を終える
- 具体的な解決策は本文で展開

---

## 🔧 提案する変更

### Before (現在のプロンプト)

```markdown
## 構成
1. 導入部: 読者の問題意識を喚起
2. 本文: 解決策を提示
3. まとめ: 次のアクションを促す
```

### After (提案)

```markdown
## 構成
1. **導入部: 読者の問題意識を喚起（3段落・300字以内）**
   - 1段落目: 読者の悩みを明示
   - 2段落目: なぜその悩みが生まれるか
   - 3段落目: この記事で何が得られるか
   - **⚠️ 導入部が長いと離脱率が上がる。簡潔に**
2. 本文: 解決策を提示
3. まとめ: 次のアクションを促す
```

**変更のポイント**:
- 導入部の具体的な長さ（3段落・300字以内）を明記
- 各段落の役割を明示して迷わせない
- 注意マークで重要性を強調

---

## ✅ 期待される効果

- エージェントが導入部を簡潔に書くようになる
- 離脱率が下がる
- 本文の展開がスムーズになる

---

## 🎯 承認後のアクション

### ARCHITECT (プロンプトエンジニア)

```markdown
# プロンプト更新提案

**提案日**: 2026-01-20
**対象エージェント**: ARCHITECT
**対象プロンプト**: prompt-design.md
**提案理由**: learning_architect_003（制約の明記が品質を安定させる）が weight 5 に到達

---

## 📊 該当する学び

### learning_architect_003: 制約の明記が品質を安定させる
- **weight**: 5
- **カテゴリ**: プロンプト設計
- **フィードバック回数**: 5回

**学んだこと**:
- 制約を明記しないと、出力がブレる
- 「やってはいけないこと」を明示することが重要
- 数値的な制約（文字数、個数など）は必須

---

## 🔧 提案する変更

### Before (現在のプロンプト)

```markdown
## プロンプト設計の原則
1. 明確な指示を与える
2. 例を示す
3. 出力形式を指定する
```

### After (提案)

```markdown
## プロンプト設計の原則
1. 明確な指示を与える
2. 例を示す
3. 出力形式を指定する
4. **制約を明記する（品質安定化の鍵）**
   - ✅ やるべきこと
   - ❌ **やってはいけないこと**（これが特に重要）
   - 数値的な制約（文字数、個数、範囲など）を明示
   - **制約がないと出力がブレる原因になる**
```

**変更のポイント**:
- 「制約の明記」を4番目の原則として追加
- 「やってはいけないこと」の重要性を強調
- 数値的制約の例を具体的に

---

## ✅ 期待される効果

- エージェントが自然と制約を明記するようになる
- プロンプトの品質が安定する
- 意図しない出力が減る

---

## 🎯 承認後のアクション

承認された場合、以下を実行します:

1. プロンプト（prompt-design.md）を更新
2. learnings.mdの learning_architect_003 の status を `resolved` に変更
3. 変更履歴をプロンプトに記録
4. resolved_at: 2026-01-20, reflected_in: prompts/prompt-design.md を追加
```

## 🔄 ARCHITECT の週次/月次レビュープロセス

このテンプレートを使用するタイミング:

### 週次レビュー（毎週月曜）

1. **各エージェントのlearnings.mdをレビュー**
   ```bash
   grep "weight: [5-9]" */learnings.md
   grep "weight: [1-9][0-9]" */learnings.md  # 10以上
   ```

2. **weight 5以上の学びを抽出**
   - status が `active` のもののみ
   - `resolved` や `archived` は対象外

3. **プロンプト更新が必要か判断**
   - 普遍的な学びか？
   - 基礎能力に関する学びか？
   - プロジェクト固有ではないか？

4. **このテンプレートを使って提案作成**
   - workspace に保存
   - ユーザーレビュー依頼

### 月次レビュー（毎月1日）

- weight 3-4 の学びも含めて検討
- 複数の学びをまとめて反映できないか検討
- プロンプト全体の見直し

## 💡 Tips

### 複数の学びを一度に反映する場合

```markdown
## 📊 該当する学び（複数）

### learning_echo_002: 文字数厳守（140字前後）
- weight: 5
- フィードバック回数: 5回

### learning_echo_003: 具体的な数字とシーン
- weight: 4
- フィードバック回数: 4回

---

## 🔧 提案する変更

これら2つの学びを「制約」セクションにまとめて反映します。
```

### ユーザーに選択肢を提示する場合

```markdown
## 🔧 提案する変更（案A）

[案Aの内容]

## 🔧 提案する変更（案B）

[案Bの内容]

**ARCHITECT の推奨**: 案A
- 理由: 〜
```
