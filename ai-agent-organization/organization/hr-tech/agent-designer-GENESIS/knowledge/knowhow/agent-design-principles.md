# エージェント設計原則

## 概要

AIエージェントを設計する際の基本原則とベストプラクティス。

---

## 1. 人格設計の原則

### 1.1 コア特性の定義

エージェントの人格は以下の3層で構成する：

```
[不変層] コア特性
    ↓
[可変層] スキル・知識
    ↓
[表層] 口調・表現
```

#### コア特性（変わらない本質）
- 2-3個の核となる特性を定義
- 価値観や行動原理を明確化
- 例: 「好奇心旺盛」「論理的思考」「共感力」

#### 強みと弱み
- 強みは3-4個、弱みは2-3個
- 弱みがあることでキャラクターに深みが出る
- 弱みは致命的でなく、愛嬌のあるものにする

### 1.2 口調・話し方

| 要素 | 設定例 |
|------|--------|
| 一人称 | 俺、私、僕、わたくし |
| 二人称 | あなた、君、お前、〇〇さん |
| 語尾 | 〜だ、〜です、〜だぜ、〜ですわ |
| 敬語レベル | カジュアル〜フォーマル |

#### 特徴的なフレーズ
- 口癖を2-3個設定
- 状況別の定番フレーズ（挨拶、承認、困った時など）

---

## 2. 役割設計の原則

### 2.1 単一責任の原則

1つのエージェント = 1つの専門領域

```
✅ 良い例: X投稿専門、note記事専門
❌ 悪い例: SNS全般担当（範囲が広すぎる）
```

### 2.2 明確な境界線

| 定義すべき項目 | 内容 |
|---------------|------|
| 担当領域 | 何ができるか |
| 対応不可 | 何ができないか（他に委譲） |
| 判断基準 | 迷った時のルール |

### 2.3 他エージェントとの関係性

- 上司・部下の関係を明確化
- 連携パターンを定義
- 会話サンプルを用意

---

## 3. 能力設計の原則

### 3.1 ステータス設計

#### 共通ステータス（全エージェント共通）
- creativity（創造性）
- accuracy（正確性）
- speed（速度）
- adaptability（適応力）

#### 専門ステータス（エージェント固有）
- 役割に応じた専門能力
- 1エージェントにつき1つ

### 3.2 スキル設計

スキルは以下の形式で定義：

```yaml
skill:
  id: "unique_id"
  name: "スキル名"
  description: "このスキルで何ができるか"
```

---

## 4. 成長設計の原則

### 4.1 成長曲線

```
初期: 急速な成長（基礎習得）
中期: 安定した成長（スキル磨き）
後期: 緩やかな成長（専門性深化）
```

### 4.2 成長トリガー

- タスク完了
- レビューフィードバック
- 新しいパターンの学習
- 失敗からの学び

---

## 5. ファイル構成

### 標準ディレクトリ構造

```
[agent-name]/
├── AGENT.md       # 人格・役割定義
├── STATUS.yaml    # 現在のステータス
├── HISTORY.md     # 成長履歴
├── prompts/       # 使用プロンプト
│   ├── task-specific-1.md
│   └── task-specific-2.md
└── knowledge/     # 専門知識
    ├── strategy-brief.md
    ├── learnings.md
    └── examples/
        ├── good-examples.md
        └── bad-examples.md
```

## 6. チェックリスト

### 新規エージェント作成時

- [ ] コア特性を3つ定義したか
- [ ] 強み・弱みのバランスは適切か
- [ ] 口調・話し方は一貫しているか
- [ ] 担当領域は明確か
- [ ] 他エージェントとの関係性は定義されているか
- [ ] 成長パスは設計されているか
- [ ] 必要なプロンプトは用意されているか
- [ ] 知識ベースの初期構造は作成されているか
