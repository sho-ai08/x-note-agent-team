# プロンプトエンジニアリング原則

## 概要

効果的なプロンプトを設計するための基本原則とベストプラクティス。

---

## 1. プロンプト構造の原則

### 1.1 基本構造

良いプロンプトは以下の要素で構成される：

```
[役割定義]
    ↓
[コンテキスト]
    ↓
[タスク指示]
    ↓
[制約条件]
    ↓
[出力形式]
    ↓
[例示（オプション）]
```

### 1.2 各要素の詳細

#### 役割定義
```
あなたは[専門性]を持つ[役割]として行動してください。
```
- 具体的な専門性を明示
- 期待される振る舞いを伝える

#### コンテキスト
```
## 背景情報
- [関連する情報1]
- [関連する情報2]
```
- タスクに必要な背景情報
- 前提条件や状況説明

#### タスク指示
```
## タスク
[具体的に何をしてほしいか]
```
- 明確で具体的な指示
- 曖昧さを排除

#### 制約条件
```
## 制約
- [制約1]: [理由]
- [制約2]: [理由]
```
- 守るべきルール
- 避けるべきこと

#### 出力形式
```
## 出力形式
[期待する出力の構造]
```
- 具体的なフォーマット
- サンプル出力

---

## 2. 明確性の原則

### 2.1 具体的な指示

```
❌ 悪い例: 良い文章を書いて
✅ 良い例: [YOUR_TARGET_AUDIENCE]向けに、[YOUR_TOPIC]について、
         500文字以内で、親しみやすい口調で書いて
```

### 2.2 数値化できるものは数値で

| 曖昧な表現 | 具体的な表現 |
|-----------|-------------|
| 短く | 100文字以内 |
| いくつか | 3つ |
| 早めに | 5分以内 |
| 詳しく | 3段落で |

### 2.3 判断基準を明示

```
## 判断基準
- 優先度: 読みやすさ > 網羅性
- トレードオフ: 詳細さより簡潔さを重視
```

---

## 3. 構造化の原則

### 3.1 階層構造を使う

```markdown
## 大見出し
### 中見出し
#### 小見出し
- 箇条書き
  - ネスト
```

### 3.2 テーブルを活用

```markdown
| 項目 | 説明 | 例 |
|------|------|-----|
| A | Aの説明 | サンプル |
```

### 3.3 コードブロックで区切る

````markdown
```
テンプレートや出力形式はコードブロックで囲む
```
````

## 4. 変数設計の原則

### 4.1 変数の命名

```
{snake_case}     # 推奨
{kebab-case}     # 許容
{CamelCase}      # 非推奨
```

### 4.2 変数リスト

| 変数名 | 説明 | 必須 | デフォルト |
|--------|------|------|-----------|
| `{topic}` | トピック | ✅ | - |
| `{tone}` | トーン | ❌ | カジュアル |

### 4.3 変数の検証

- 必須変数が未入力の場合の処理
- デフォルト値の設定
- 許容値の範囲

## 5. 例示の原則

### 5.1 Few-shot Learning

良い例を2-3個示すことで、期待する出力を明確化：

```
## 例

### 入力例1
[入力]

### 出力例1
[期待する出力]

### 入力例2
[入力]

### 出力例2
[期待する出力]
```

### 5.2 良い例と悪い例

```
## 良い例
✅ [良い出力例]
理由: [なぜ良いか]

## 悪い例
❌ [悪い出力例]
理由: [なぜ悪いか]
```

## 6. テストと改善

### 6.1 テスト項目

- [ ] 期待通りの出力が得られるか
- [ ] エッジケースでも動作するか
- [ ] 制約条件が守られているか
- [ ] 出力形式が一貫しているか

### 6.2 改善サイクル

```
設計 → テスト → 分析 → 改善 → 設計...
```

### 6.3 バージョン管理

- 変更内容を記録
- 前バージョンとの比較
- 効果測定
