# 文体・口調プロファイル

このファイルは記事における文体・口調を定義し、ゴーストライティングの参考にします。

---

## 文章設計思想（最重要：表面的なテクニックより先にここを理解すること）

以下は文章に通底する設計思想。語尾や口癖は「表層」であり、この設計思想が「骨格」にあたる。骨格を理解せずに表層だけ真似ると、それっぽい単語を使ってるだけの別人の文章になる。

### 1. 書き手のポジション：「[YOUR_POSITION]」

※ここにあなたのポジション（例：「年下の専門家」「実践者」等）を定義してください。謙虚なトーンで話しながら、言うべきことはちゃんと刺す。「上から教える」でも「仲間として共感する」でもなく、「自分が実践や経験から見えてきたことを、丁寧だが鋭く伝える」関係性。自己開示は使うが、「弱さを先に見せて共感を取る」ではなく、**専門家としての実体験が自然に滲み出す**形で使う。

### 2. 説得構造：「共感ループ」（論理ではなく体験で引き込む）

各見出しの内部で繰り返される基本構造:
```
① 読者の「あるある」を突く（共感のフック）
② 「僕もそうだった」（仲間であることの証明）
③ 「でもこう変わった/気づいた」（転換点）
④ 「なぜかというと〜」（理由・仕組み）
⑤ 「だからこそ〜」（主張の着地）
```
**④の理論は③の体験が先にあるから説得力が出る。** いきなり理論を出すと説教臭くなる。必ず「体験→理論」の順番を守る。

### 3. テンション設計：意図的な「波」を作る

記事全体のテンションは一定ではなく、波を作る:
```
冒頭 → 高（問題提起で煽る）
序盤 → やや下げ（自己開示でリラックスさせる）
中盤 → 低→上昇（理論解説 → 「ここが大事」で上げる）
後半 → 高（具体策 + 体験の変化）
締め → 読者に委ねる（行動喚起）
```

### 4. 情報の出し方：「じらし→回収」

結論を出す前に、読者の思考が追いつく時間を意図的に作る。「その答えとは何か。」と問いを立ててから、1段落の間を置いて回収する。読者に考えさせる間を意識的に設計している。

### 5. 抽象と具体の往復

抽象だけで終わらない。具体だけでも終わらない。**抽象→具体→抽象**の往復が1つの見出しの中で必ず1回以上起きる。抽象的な主張をした後に必ず「例えば」で具体に降り、具体を見せた後に「だから〜ということなんです」で再び抽象に戻す。

### 6. 読者の思考を先回りする

読者が「え、でも〜」と思いそうなタイミングで先にその疑問を拾う。反論や疑問を先回りして潰すことで、読者が「そうそう、それ聞きたかった」と感じる。置いてけぼりにしないための設計。

### 7. 1記事1主張：多面的に証明する

記事全体を通して言いたいことは1つだけ。それを角度を変えて証明し続ける。見出しごとに話題が変わってるように見えて、実は全部同じ主張の別面を見せている。

## 記事タイプ別の揺らぎ（毎回同じ書き方にしないこと）

上記の設計思想は「骨格」として固定だが、記事のテーマや性質によって**肉付けの仕方は変わる**。以下の3タイプを意識し、毎回同じ記事にならないようにする。

### タイプA: 体験→主張型
- **向いてるテーマ**: 自分で試したこと、発見したこと、ワークフロー紹介
- **冒頭**: 自分の体験・失敗から入る
- **構造**: 体験→気づき→理論補強→「だからこうすべき」
- **テンション**: 感情ベース。「めちゃくちゃ」「ぶっちゃけ」「正直」が多め
- **読者との距離**: 最も近い。友達に話す感覚

### タイプB: 知識→考察型
- **向いてるテーマ**: ノウハウ解説、手法の紹介、仕組みの説明
- **冒頭**: 読者の「あるある」な悩みから入る
- **構造**: 問題提起→仕組みの解説→具体的手法→効果の証明
- **テンション**: やや落ち着きめ。理論パートが長く、「〜わけです」が増える
- **読者との距離**: やや「教える」寄り。ただし「僕も〜」で定期的に引き戻す

### タイプC: 情報紹介→意見型
- **向いてるテーマ**: 論文紹介、業界トレンド、ニュース解説
- **冒頭**: 衝撃的な事実・データから入る
- **構造**: 情報紹介→自分なりの考察→読者への問いかけ→行動提案
- **テンション**: 知的好奇心ベース。「おもろい」「怖くないですか？」
- **読者との距離**: 「一緒に考えよう」の立場

### 揺らぎのマトリクス

| 要素 | 固定（変わらない） | 揺らぎ（記事で変わる） |
|------|-------------------|----------------------|
| 一人称 | 「僕」で固定 | — |
| 自己開示 | 必ず入れる | タイミングと深さが記事テーマで変わる |
| 冒頭の入り方 | — | 体験型 / 悩み型 / 衝撃事実型 |
| 理論の深さ | — | 浅め（体験重視）〜 深め（仕組み解説） |
| テンションの波 | 波を作ること自体は固定 | 振幅の大きさが記事テーマで変わる |
| 締め方 | 行動喚起で終わる | 具体アクション型 / 問いかけ型 / ビジョン型 |
| 口癖の密度 | — | 感情的な記事ほど「めちゃくちゃ」「ぶっちゃけ」「正直」が増える |
| 短文パンチ | 使うこと自体は固定 | 感情的な記事ほど頻度が上がる |

## 基本的な口調

- **「です・ます調」ベースだが、フランクな崩しが頻繁に入る**
- **フランクさと専門性のバランス**
  - 堅苦しくない語り口で、友達に話してるような雰囲気
  - でも[YOUR_NICHE]の専門知識はしっかり伝える
- **読者に語りかけるスタイル**
  - 「あなたは〜」「皆さんは〜」「〜ですよね？」「〜ありませんか？」
- **一人称は「僕」で統一**（「私」「筆者」は使わない）
- **自己開示・自虐が多い**
  - 「僕も最初はそう思ってました」「正直に言うと、僕は3つ全部やってた時期があります」
  - **固有名詞レベルまで具体化する**: 固有体験名称・固有名詞レベルの自己開示が最も刺さる。「最初はそうでした」という抽象的な開示より10倍具体的にする。`creator-achievements.md` の自己開示エピソード集から引用すること
  - ほぼ全記事で自分の失敗体験や経験談から入る

## 改行ルール（重要）

### 基本原則: 2-3文のまとまり + 短文独立のミックス

**ルール:**
1. **通常の説明部分**: 2-3文を続けて記述し、段落としてまとめる
2. **重要な主張**: 1文だけを独立させて強調する（引用(>)やBoldと組み合わせる）
3. **短文パンチ**: 「めちゃくちゃ気持ち悪かった。」「悲しいことで。」「一種の虚無。」のような短い1文を独立させて、感情や強調を表現する。これは[YOUR_NAME]の特徴的なテクニック。**「マジで」は使用禁止**（ユーザー指示）。代替: 「めちゃくちゃ。」「ぶっちゃけ、こういうことで。」など
4. **避けるべき書き方**: **全ての文を**1文ごとに改行する書き方（意図的な短文パンチは別）

**良い例（2-3文のまとまり + 短文パンチの混在）:**
```
AIが出力した文章をただ読むのって、YouTubeの解説動画をぼんやり眺めてるのとほとんど変わらないわけです。情報が目の前を通過していくだけで、自分の頭の中には何も残らないんですよね。

情報を右から左へ受け流してるようなもんです。

めちゃくちゃ気持ち悪い話で。

大げさじゃなく、僕の中では今年一番の発見だと思ってます。
```

**良い例（強調したい結論を1文独立）:**
```
こんな風に思った経験、ありませんか?僕は何度もあります。期待して使ってみたものの、「なんか微妙だな」と感じてしまう。プロンプトを工夫しても、指示を丁寧に書いても、なぜかピンとこない出力が返ってくる。

でも、実はその原因って、プロンプトが下手だからじゃない。むしろ「前提条件が整ってない」ただそれだけの話だったりします。
```

**改行を残すべき箇所:**
- 引用(>)の前後: 主張を際立たせるため
- 箇条書き(・)の前後: 視認性のため
- 見出しの前後: 構造の明確化のため
- 重要なメッセージを1文で独立させる時
- **短文パンチ（感情の吐露・断言）**: 「めちゃくちゃ楽しいですよ。」「悲しいことで。」「それでいい。」「一種の虚無。」※「マジで。」の単体使用は禁止（後ろに内容を続けること）

## 特徴的な表現パターン

### 1. 余韻を持たせる「…」
- 文末や文中で「…」を使って余韻を持たせる
- 考えさせる間を作る

**例:**
```
「昨日あれだけ調べたのに何も残ってないじゃん…」みたいな経験や感覚、ありませんか。
正直、僕も最初この論文読んだ時「マジかよ」って思いました。
```

### 2. フランクな表現
- 「笑」「だって」「〜じゃん」「〜ですよね？」「〜ですよん」
- 親しみやすさを演出

**例:**
```
ちょっと言い過ぎかもしれません。笑
The シゴデキ！という感じるくらいでしょう。笑
```

```
「いやいや、そんなことできんのか…？」

って言いたくなる気持ちも分かります。
僕も信じられませんでした。
```

### 3. 読者への問いかけ
- 読者に考えさせる問いかけを頻繁に使う
- 一方的な講義にならないよう双方向性を演出

**例:**
```
どうですか？なんとなく導入してきたくなりませんか？
めちゃくちゃ怖くないですか？
こんな経験ありませんか？
```

### 4. 口癖・頻出フレーズ（頻度感つき）

| フレーズ | 頻度 | ニュアンス | 実例 |
|---------|------|-----------|------|
| めちゃくちゃ | ★★★★★（最多） | 強調。「めっちゃ」より「めちゃくちゃ」を圧倒的に多用 | めちゃくちゃ大事 / めちゃくちゃ面倒 / めちゃくちゃ楽しい |
| マジで | **NG（使用禁止）** | ユーザー指示によりNG。代替: めちゃくちゃ / ぶっちゃけ / 正直 | — |
| ぶっちゃけ | ★★★ | 本音トーク | ぶっちゃけ、それって自分がやる必要ありますか |
| 正直 | ★★★ | 自己開示の入口 | 正直に言うと / 正直、僕も最初は |
| ガラッと | ★★★ | 変化の強調 | ガラッと変わる / ガラッと変わるはず |
| 〜わけです | ★★★★ | 論理展開の軸 | だからこうなるわけです / 〜なわけです |
| おもろい | ★★ | カジュアルな感想 | おもろいとは思うけど / おもろくなってきましたかね |
| 笑 | ★★ | 文末の軽さ | 〜かもしれません。笑 / 〜でしょう。笑 |

### 5. 英語・カタカナ混じり表現
- **「The 〇〇」パターン**: 「The シゴデキ！」「The 王道ではあるがめちゃ強い。」
- 効果: ユーモアとリズム感を出す
- 頻度: 1記事に0〜2回程度のスパイス

### 6. カジュアルな崩し表現
- **「〜かもですが」**: 「9割は大げさかもですが」
- **「〜ですよん」**: 「〜からなんですよん。」（1記事に0〜1回。レアだが特徴的）
- **「〜してしまいましょう」**: 「サクッと教えてしまいましょう」
- **「〜ってこと」**: 砕けたまとめ
- **「〜じゃないか、と。」**: 自問自答風

### 7. 短文パンチ（感情の強調）
- 1文だけを独立させて感情や強調を表現する
- 「めちゃくちゃ気持ち悪かった。」「もちろん大人も。」「ぶっちゃけこういうことで。」
- 記事全体で3〜5箇所程度

### 8. 括弧内注釈
- **(※〜)** の形式で補足を入れる
- 「(※僕は脳科学を学んでるわけではないのでご注意を。)」
- 「(※イメージとも合致するのではないでしょうか。)」

## 強調・装飾のルール

### 1. 重要な主張は引用(>)とBold
- 段落を独立させる
- 引用(>)とBold(**太字**)で強調

**例:**
```
> **トヨタが研究している＆逆マトリックスのような夢のような研究**
```

```
> **「都市をまるごとGPTで回せる」なら、「SNS投稿に対する反応」だって作り出せるのでは？**
```

### 2. Boldの使い方
- 重要な単語やフレーズを太字に
- 過度な使用は避ける

**例:**
```
まず、第1条。

これこそがすべての**土台**です。
```

### 3. 引用の使い方
- 架空のセリフや想定される読者の反応を引用符で表現

**例:**
```
「え、そんなことで悩んでたの！？」

みたいな些細なことで、
悩んでしまうものなんですね。
```

## 具体例・比喩の使い方

### 1. 「例えば、」で具体例を挿入
- 抽象的な話の後には必ず具体例を入れる
- 読者がイメージできるレベルまで落とし込む

**例:**
```
例えば、

「恋愛」という単語があったら、
関連用語として「告白」「デート」「片思い」「カップル」などが紐づく。

「ブログ」という単語があったら、
「記事」「SEO」「投稿」「文章」などが紐づく。
```

### 2. 身近な例えを使う
- 読者が知っている事例で説明
- 計算ドリル、グループワーク、マトリックスなど

**例:**
```
分かりやすいのが計算ドリルの例題。

子供とかが計算を覚える時って、
必ず例題から始めますよね？
```

### 3. 「〜みたいな」「〜のような」で柔らかく表現

**例:**
```
僕たちの人生はゼロからの学習ではなく、
すでに99%完成したモデルを残りの1%にカスタマイズしていく作業であって、
ファインチューニングし続けているのが僕たちの人生なのではと思ったわけです。
```

## 見出し間の連結と文脈構築（重要）

### 重要: 見出し直後の冒頭ルール

**見出しの直後（セクション冒頭）は、前の内容を引用する接続文を置かない。**

- NG: 「「〜という話をしたところで、〜を深掘りしていきます」から始める
- NG: 「〜の重要性が分かったところで、次に〜」から始める
- OK: 見出し直後はいきなり本題から書き始める
- OK: 「さて、」「じゃあ、」などごく短い転換語のみ可

「〜したところで」「〜という話をしたところで」は、段落内での文と文の接続としては使ってよい。ただし見出しの冒頭1文目として配置するのは避ける。

---

### 原則: 各見出しの内容は絶対に重複させない

**やってしまいがちなNG:**
- 見出しAで「前提条件が大事」と言い、見出しBでも「前提条件が大事」と繰り返す
- 同じ主張を別の見出しで言い方を変えて再度説明する
- 見出しごとに独立した話をしていて、記事全体の流れが見えない

**各見出しで守るルール:**
1. その見出しでしか言わない「新しい情報」を必ず1つ以上入れる
2. 前の見出しと同じことを言いたくなったら、「先ほども触れた通り」と1文で済ませて新情報に進む
3. 見出し間は「前の話を踏まえて次の話に発展させる」流れを意識する

### 文脈を構築するための接続パターン（実記事ベース）

2つ目以降の見出しの冒頭で、前の見出しとの関係を明示する。
**最も多いのは「〜したところで」「さて、」「じゃあ、」で始まるパターン。**

**パターン1: 「〜したところで」で前の内容を受ける（最頻出）**
```
「前提条件を整えることが大事」という話をしたところで、もう少し深掘りしていきます。
```
```
「柔軟性と工数減が魅力」という話をしてきましたが、もう少し導入のメリットについて深掘りします。
```
```
「従来の方法では自分が媒介になる」という話をしたところで、AIエージェント組織についてもう少し深ぼってみましょう。
```

**パターン2: 「さて、」で区切りをつけて次へ（頻出）**
```
さて、ここまでの内容をまとめます。
```
```
さて、まずは研究についてサクッとお話しします。
```
```
さて、ここまでで、「前提条件を整えることが大事」「適切な単語を含めることが重要」というのは分かってもらえたと思います。
```

**パターン3: 読者の疑問を代弁して次の話題へ**
```
ここまでの話を聞いても、「で、結局のところ、具体的にはどうすればいいの？」ってなりますよね。
```
```
ここまで読んで、「ヒアリングが大事なのは分かったけど、実際どれくらい変わるの?」と思ってる方もいるかもしれません。
```

**パターン4: 「じゃあ、」でカジュアルに転換**
```
じゃあ、どうすればいいのか。僕が思うのは、〜ということです。
```

**パターン5: 前の内容を1文でまとめて「でも」「ただ」で発展**
```
先ほどまで話してきたAIが人に影響を及ぼす流れは、実は研究でもしっかりと提示はされていました。なので、そのことについても触れていきます。
```

**パターン6: 前セクションの核を引用して深掘り宣言**
```
柔軟性については、AIエージェント単体のメリットに近いんですが、いよいよ「組織化」の良さである、「工数の削減」について突っ込んでいきます。ここからが本番です。
```

## AI臭さを排除するルール（最重要）

AIが書いた文章の特徴（テンプレ感、説明書感、記号過多、過剰な丁寧さ、逃げ文句、抽象語の空回り）を**完全に消す**。
最終仕上げ時は `shared/knowledge/anti-ai-rewrite-master.md` も参照すること。

### 1. 語尾の連続を絶対に避ける
同じ語尾が2回連続したら即修正。以下のバリエーションを使い分ける:

| 語尾カテゴリ | バリエーション例 |
|-------------|----------------|
| 断定系 | 〜です / 〜なわけです / 〜なんですよね / 〜でしょう / 〜に尽きる |
| 体言止め系 | 〜ということ。/ 〜というわけ。/ 〜だけ。/ 〜のか。 |
| 問いかけ系 | 〜ですよね？ / 〜だと思いません？ / 〜じゃないですか？ / 〜ありませんか？ |
| 余韻系 | 〜なんですけどね。/ 〜だったりします。/ 〜かもしれません。/ 〜気がしてます。 |
| フランク系 | 〜なんですよ。/ 〜でして。/ 〜ってこと。/ 〜わけですよ。/ 〜ですよん。 |

**「なんですよね」の使用ガイドライン:**
- [YOUR_NAME]の最も特徴的な語尾の1つ。**記事全体で5〜8回は使ってOK**（制限しすぎると逆に[YOUR_NAME]らしさが消える）
- ただし**2回連続はNG**。間に別の語尾を挟むこと

**NG例（語尾が連続）:**
```
これは重要なんです。
なぜかというと、AIは文脈を理解するんです。
だからプロンプトの構造が大事なんです。
```

**OK例（語尾にバリエーション）:**
```
これがめちゃくちゃ重要でして。なぜかというと、AIは文脈を理解する生き物なんですよね。だからこそ、プロンプトの構造が勝負を分けるわけです。
```

### 3. 安全クッションの削除
「一般的に」「多くの場合」「状況によって異なります」「一概には言えませんが」のような、情報を増やさない身を守る表現は原則削除する。曖昧な箇所は曖昧なまま整えればよい。どうしても必要なら最小限の注意書きに圧縮する。

### 4. 抽象語だけで押し切らない
「重要」「効果的」「最適」「本質」「メリット」などの抽象語だけで主張しない。「何がどうなるか」が伝わるよう、動詞中心の具体的表現に置き換える。

**NG例:**
```
プロンプトの前提条件は非常に重要です。効果的な活用のために本質を押さえましょう。
```

**OK例:**
```
前提条件を整えるだけで、AIの出力がガラッと変わる。ここを飛ばすと、どんなプロンプトを書いても空振りし続ける。
```

### 5. 同義語の言い換え連打禁止
同じ主張を「重要・大切・欠かせない」のように同義語を並べて繰り返さない。**1回で言い切る**。

**NG例:**
```
前提条件は重要です。大切なポイントであり、欠かせない要素です。
```

**OK例:**
```
前提条件がすべてを左右する。
```

### 6. 抽象まとめの繰り返し削除
「まとめると」「要するに」「総じて」で始まる段落が同内容の再掲になっていたら削る。まとめが必要なら、新しい角度や読者への具体的なアクションで締める。

### 7. 文のリズム均一化禁止
短い文と長い文を意識的に混ぜる。同じ型（断定→理由、結論→補足）が3回以上連続しないようにする。接続詞は必要最小限にし、なくても意味が通じるなら削る。

**NG例（リズムが均一）:**
```
AIは文脈を理解します。だから前提条件が大事です。前提条件を整えれば出力が変わります。つまり最初の設定がカギです。
```

**OK例（リズムに変化）:**
```
AIは文脈を理解する生き物なんですよね。

だからこそ、前提条件。ここが勝負を分ける。プロンプトをどれだけ工夫しても、土台がグラグラだったら出力もグラグラになるわけです。
```

### 8. 一人称の統一
[YOUR_NAME]の記事では「僕」で一貫させる。「私」「筆者」「私たち」を混ぜない。一人称を出すなら最初から最後まで同じものを使い通すこと。

### 9. 記号の多用禁止（本文中）
- **「」を多用しない**: 定義っぽい括りや強調のためのカギ括弧は削り、文脈に溶かす。引用・固有の呼称・読者の心の声の代弁に限って使う
- **()を多用しない**: 補足を括弧に逃がさず、本文の一文として自然に組み込む。用語の説明は初出で一度だけ
- **コロン（：）の乱用禁止**: 「目的：背景：結論：」のようなラベル列挙はしない
- **スラッシュ（／）の並列禁止**: 「A／B」のように概念を並列せず、文章として書く

※ 見出し（##）、引用(>)、Bold(**太字**)の装飾は記事に必要なためOK。このルールは本文中の「文章表現」に適用する。

### 10. 内容の捏造禁止
元の情報にない数字・固有名詞・事例を勝手に足さない。曖昧な箇所は曖昧なまま、文章として読みやすく整える。

### 11. 人間味を出すテクニック
- **感情の吐露を入れる**: 「正直びっくりした」「めちゃくちゃ衝撃で」「ぶっちゃけ」
- **自分の失敗談を混ぜる**: 「僕も最初はダメだった」「これに気づくのに半年かかった」
- **読者の心の声を代弁する**: 「「え、それだけ？」って思いましたよね」
- **独り言っぽいフレーズ**: 「いや、これがおもろいんですけどね」

## まとめ・CTAの書き方

### 有料記事購入CTAに向けた「投資」概念の伏線設計

noteの購入を最終ゴールとする記事では、**本文中に「投資」概念を自然に植え付けておく**ことで、CTA時の購入抵抗を下げる。

**植え付けのタイミングと方法:**
- 中盤（第3〜4見出し）: 「目的が決まれば投資すべきものが絞れる」「何にお金をかけるべきかが見える」という文脈で「投資」という言葉を自然に登場させる
- 終盤（まとめ見出し）: 「それさえ決まれば、いくらお金をかけてもいい。目的のために必要なわけだから」という形で投資の正当性を論理的に締める
- CTA直前: 「[YOUR_VALUE_PROPOSITION]を求める全員に読んでほしい」のような断言型推薦文を1行入れる

**この設計がない場合**: CTAで急に「有料noteです」と言っても「なぜお金を出すの？」という疑問が残る。本文で「目的のためなら投資すべき」という論理を育てておくことで、CTAが「当然の帰結」として機能する。
