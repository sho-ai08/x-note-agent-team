# note記事作成プロセス

このファイルはnote記事作成の標準プロセスを記載します。

---

## 基本プロセス（ステップバイステップ）

### Step 1: 見出し構成作成

入力された元情報を詳細に分析して、漏れなく情報を伝えるための見出し構成を設計する。

**重要ポイント:**
- 各見出しは1,000字程度のボリュームを想定して適切に情報量を分割
- 見出し名の口調は以下の参考例をトレースしながら、元情報に合わせた見出し名を提案

**見出し名の参考例（単語の流用は禁止）:**
```
## コンセプトは「トレンド×普遍」が最も生き残る
## 人は常に新しい刺激を渇望している生き物
## 自分と読者の読後感・イメージをピタッと揃えよ
## プロンプトは「自分の脳」をAIに移植するつもりで書くべし
## 人がモノを買うのは「欲」と「コンプレックス」でしかない
## 人を魅了するには「余白」を自在に操ること
```

**出力形式:**
- コードブロック形式で出力

#### Step 1.5: 見出し別コンテンツマップ作成（重複排除）

見出し構成が決まったら、**各見出しで「何を書くか」「何を書かないか」を明確にするコンテンツマップ**を作成する。

**コンテンツマップの書き方:**
```
## [見出し1のタイトル]
- この見出しで伝える新情報: [具体的に]
- この見出しでは触れない内容: [他の見出しで扱うもの]
- 前の見出しとの接続: なし（冒頭）

## [見出し2のタイトル]
- この見出しで伝える新情報: [具体的に]
- この見出しでは触れない内容: [見出し1で既に述べたもの]
- 前の見出しとの接続: [見出し1のどの内容を踏まえて発展させるか]

## [見出し3のタイトル]
- この見出しで伝える新情報: [具体的に]
- この見出しでは触れない内容: [見出し1, 2で既に述べたもの]
- 前の見出しとの接続: [見出し2のどの内容を踏まえるか]
```

**コンテンツマップ作成時の鉄則:**
1. 各見出しの「新情報」が他の見出しの「新情報」と被っていないか確認する
2. 前の見出しの内容に言及する場合は「〇〇でも触れた通り」程度に留め、改めて説明し直さない
3. 記事全体が「1本の流れ」になっているか確認する（見出し1→2→3で読者の理解が段階的に深まるか）

---

### Step 1.7: リード文作成

見出し構成とコンテンツマップが完成したら、リード文を作成する。

**参照:** `lead-text-patterns.md` のパターン集から、記事テーマに最適な型を選択する。

**リード文作成のルール:**
1. 直近3記事と同じリード文パターンを避ける
2. `lead-text-patterns.md` の10個の型から選ぶか、複数の型を組み合わせる
3. テンプレートの丸写しは禁止（構造だけ参考にし、言い回しは毎回変える）
4. 最初の1文で読者の手を止めること（スクロールストッパー）

---

### Step 2: 見出し作成（1見出しごと）

Step1で設計した見出し構成をもとに、note参考例をフォーマットとして参照して、1見出しずつ書く。

**執筆ルール:**
1. **1つの見出しの文章生成が終われば出力を強制的に終了する（絶対厳守）**
2. 口調プロファイルを詳細に読み込み、各見出しの文章をゴーストライティングする
3. 元情報の口調は引き継がない
4. 文章のみで各見出しの文章を書く（1,000文字以上を遵守）
5. 重要な主張文だけは段落を独立させ、「引用(>)&Bold」のMarkdownのみで強調
6. **【最重要】各見出しで内容を重複させない。Step 1.5のコンテンツマップに従い、各見出しの「新情報」のみ書く。前の見出しの内容を繰り返す場合は「先ほども触れた通り」程度に留める**
7. **2つ目以降の見出しの冒頭では、前の見出しの内容を踏まえて自然に連結する**（`writing-style-profile.md`の「見出し間の連結と文脈構築」のパターンを使用）
8. 各見出し作成の途中では、必ず「読者の脳内にイメージができるレベル」の例えや具体例を提示
9. 単独の接続詞を2,3箇所以上配置（読者の休憩エリアと文章間の連結を実現）
10. **【重要】改行ルール**: 2-3行で固める、主張だけ1文独立（詳細は`writing-style-profile.md`の「改行ルール」を参照）
11. **【重要】AI臭さ排除**: 語尾の2回連続禁止、NG表現リストの使用禁止（詳細は`writing-style-profile.md`の「AI臭さを排除するルール」全11項目を参照）。加えて以下も厳守:
    - 安全クッション（「一般的に」「多くの場合」「一概には言えませんが」）を削除する
    - 抽象語（「重要」「効果的」「本質」）だけで押し切らず、動詞中心の具体表現にする
    - 同義語の言い換え連打（重要・大切・欠かせない）をせず、1回で言い切る
    - 短い文と長い文を混ぜ、文のリズムを均一にしない
    - 一人称は「僕」で統一（「私」「筆者」を混ぜない）
    - 「」・()・：・／ を本文中で乱用しない
12. **人間味を出す**: 感情の吐露、失敗談、読者の心の声の代弁を各見出しに最低1つ入れる

**使用可能なMarkdown:**
- ##（見出し）
- 引用(>)
- Bold（**太字**）
- その他のmarkdownは使用禁止

**出力フォーマット:**
```
## [見出し名]
[本文]
```

---

## 絶対条件

- Markdownは「##, 引用(>), Bold」のみ使用許可
- 指示の復唱や余計な補足は入れない
- note参考例の文章を流用することを禁止（1文でも使用した場合は強制リトライ）

---

## note参考例から学んだ文体の特徴

### 1. リード文の特徴
- 冒頭で読者をグッと引きつける「型」を使い分ける
- 毎回同じパターンにならないよう、`lead-text-patterns.md`の10個の型から選択
- 読者の問題指摘、理想の未来提示、意外性、ストーリーなど多彩なバリエーション
- 最初の1文で読者の手を止めること
- **詳細は`lead-text-patterns.md`を参照**

**例（問題指摘型）:**
```
プロンプトを工夫しても、なぜか思い通りの出力が返ってこない。指示を丁寧にしても、具体的にしても、どこか微妙な結果になってしまう。

実はその原因、プロンプトの書き方じゃなくて「前提条件が整っていない」ことにあったんです。
```

**例（権威・実績型）:**
```
こんにちは、[YOUR_NAME]です。普段Xで発信してる僕ですが、本業は「[YOUR_EXPERTISE]」なんですよね。その中で、最近おもしろい発見をしたんです。
```

### 2. 本文の特徴
- 短い文を多用（1-2行で改行）
- 接続詞を単独行で配置して読者に休憩を与える
  - 「さて、」「では、」「そして、」「でも、」「つまり、」
- 読者に語りかける形式（「あなたは〜」「皆さんは〜」）
- 「…」で余韻を持たせる
- 「笑」「だって」などフランクな表現を適度に使用

**例:**
```
さて、

ここからやっと本題に入っていきますが、

そのまま各項目を説明しても、
「結局、この10ヶ条を学んだらどうなるの？」
というのがフワッとしちゃうと思うので、

「最終ゴールはどこなのか」を明確にしておきます。
```

### 3. 強調の使い方
- 重要な主張を独立段落にして引用(>)とBoldで強調

**例:**
```
> **トヨタが研究している＆逆マトリックスのような夢のような研究**
```

### 4. 具体例・比喩の使い方
- 「例えば、」で具体例を挿入
- 身近な例えを使って理解を助ける
- 「〜みたいな」「〜のような」で柔らかく表現

**例:**
```
分かりやすいのが計算ドリルの例題。

子供とかが計算を覚える時って、
必ず例題から始めますよね？

例題を見せれば、同じように解けるようになる。
```

### 5. まとめ・CTA
- 記事の最後に要点を整理
- 読者へのメッセージを添える
- [YOUR_UPSELL_PRODUCT]への自然な誘導

**例:**
```
### 少しお知らせ

[YOUR_AUTHORITY_BASIS]の僕が、

**表では話せない「[YOUR_EXCLUSIVE_CONTENT]」**
をぶちまける[YOUR_UPSELL_PRODUCT_NAME]、やってます。
```
